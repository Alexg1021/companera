The context

The promotora is the product

Zócalo Health's 85% monthly retention isn't an algorithm — it's a person. The promotora de salud is the reason hard-to-reach Medicaid members stay engaged, keep appointments, and avoid preventable ER visits. She's the bridge between clinical plans and the realities of daily life in the community.

Most care coordination software is designed for a desk, not a doorstep. It assumes English, connectivity, and a clinical workflow that doesn't match how community health workers actually operate. Compañera is built around a different premise: the tool should serve the promotora, not the other way around.

The thinking

What the gap looks like

A promotora manages 15–20 members across scattered neighborhoods, communicating primarily via WhatsApp, tracking follow-ups in her head or a notes app, and relying on institutional memory that doesn't survive staff transitions. When a member goes quiet, there's no system that flags it.

The gap
No tool built for the field

Existing EMRs and care platforms are desktop-first, English-only, and optimized for clinical billing — not community outreach. Promotoras work around them, not with them.

The opportunity
Amplify what's already working

A lightweight, mobile-first, Spanish-first companion that shows who needs outreach, makes logging a touchpoint take 30 seconds, and surfaces the right context at the right moment.

What was built

A working prototype, not a mockup

Compañera is a full-stack web app with a real Postgres database, role-based auth, and live data flowing through every screen. The demo credentials below let you log in as a promotora and walk through an actual workflow end to end.

🎯
Server-side triage
Members are automatically prioritized by urgency — no contact in 7+ days, post-ER without follow-up, upcoming Medicaid renewal. Computed on the server and updated instantly when a touchpoint is logged.
📋
Touchpoint logging
Log a WhatsApp, call, home visit, or clinic interaction in under 30 seconds. Notes are bilingual. Outcome updates the member's status and timeline immediately via revalidatePath.
🚨
Clinical escalation
One toggle on the log form creates a notification record for the clinical team. The pattern is wired through the DB with RLS — it's the foundation for a real notification system.
📊
Payer dashboard
A separate role-gated view for health plan partners showing retention, HEDIS metrics, and active plan partnerships. Active member count pulls from live DB data.
🌐
English / Spanish toggle
Full i18n across all UI surfaces. Every static string is keyed through a locale dictionary — switch languages in Settings and the entire app responds immediately.
📱
Mobile-first, installable
Built for a 390px phone screen with safe-area insets and 44px tap targets. Includes a PWA manifest — add to home screen on any device and it runs like a native app.

Under the hood

Architecture decisions worth noting

The choices below aren't accidental — each one reflects a real constraint or future need of a system like this.

Triage on the server lib/triage.ts

Urgency logic runs server-side and persists to the DB on each touchpoint save. The client never computes it. This keeps the list consistent across devices and makes future rule changes a single edit.

Role model in the DB users.role

A role column on every user record drives routing at the middleware level. Promotoras, clinicians, and payers each land in the right context — built for a multi-role system from day one.

RLS throughout Supabase policies

Row Level Security means a promotora can only see her own members. The anon key is safe to expose in the browser because the DB itself enforces access — no server-side secret key required.

Notifications as a pattern notifications table

Escalations don't just set a flag — they write a row to a notifications table with member linkage and author. It's a foundation, not a workaround.
